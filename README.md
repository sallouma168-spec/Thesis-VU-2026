# Judging the Judge — Reproducibility Guide

Code and data for the BSc Artificial Intelligence thesis:

**"Judging the Judge: A Retrieval-Augmented Audit of Directional Hallucinations in LLM Political News QA"**
Selma Ghedas (2756926) · Vrije Universiteit Amsterdam · Better AI
Supervisor: Dr. Jieying Chen · Second reader: Giacomo Zamprogno

This guide is written for the examiner and second assessor. It explains how to
set up the environment, re-run the pipeline, and verify each result and figure
reported in the thesis.

---

## 1. What this repository contains

A Naive-RAG measurement pipeline that tests whether prompt framing (neutral /
positive / negative) changes the rate and ideological direction of LLM
hallucinations when answering grounded questions about political news articles
from the **QBias** corpus. Four model configurations were evaluated
(GPT-4o-mini, DeepSeek V3.2, Llama 3.1 70B, Llama 3.3 70B).

The central result is methodological: an initial LLM-as-judge produced a
spurious framing effect (p = 0.020) that turned out to be a measurement
artifact. After the judge was corrected, no significant framing effect remained
for any model. **Both the buggy and corrected judge are included so the
discovery can be reproduced** (see Section 6).

The numbered scripts (`01_` … `07_`) follow the five-stage pipeline shown in
**Figure 1** of the thesis:

| Thesis pipeline stage | Script(s) | Output |
|---|---|---|
| Filter QBias to election articles | `01_filter_election_articles.py` | `data/election_articles.csv` |
| Balanced sampling (999 / batch) | `02_sample_articles.py`, `sample_batch2.py`, `sample_batch3.py` | `data/sampled_articles*.csv` |
| 1. Question generation | `03_generate_questions*.py` | `data/articles_with_questions*.csv` |
| 2. Framing (neutral/pos/neg) | `04_apply_framing*.py` | `data/framed_prompts*.csv` |
| 3. Grounded answering | `05_rag_answer_*.py`, `05b_rag_logprobs.py` | `data/rag_responses_*.jsonl` |
| 4. Hallucination judge | `06_judge_hallucination*.py`, `06_judge_rq3_corrected.py` | `data/hallucination_labels_*.jsonl` |
| 5. Analysis | `make_rq1_charts.py`, `rq3_regression_corrected.py`, `07_build_coding_sheet.py` | figures, tables, coding sheet |

## 2. Setup

Requirements: **Python 3.10+** (developed on 3.11). **R 4.x** is optional — only
needed to re-render the final figures with `scripts/thesis_pipeline.R`.

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### API keys

The answering and judging scripts call four hosted APIs. Copy the template and
fill in your own keys:

```bash
cp .env.example .env
# then edit .env and paste your keys
```

`.env` must contain:

```
OPENAI_API_KEY=...       # GPT-4o-mini (answering, judging, logprobs)
DEEPINFRA_API_KEY=...    # DeepSeek V3.2, Llama 3.1/3.3 70B
GROQ_API_KEY=...         # Llama 3.3 70B (provider cross-check)
MISTRAL_API_KEY=...      # only for excluded exploratory runs (optional)
```

All scripts load keys with `python-dotenv`; **no keys are hardcoded**. Run every
script from the **project root** (`thesis_benchmark/`) so the relative
`data/...` paths resolve, e.g. `python scripts/05_rag_answer_openai.py`.

> **Note on exact reproduction.** All model calls use `temperature=0` for
> determinism, but hosted LLM endpoints can still return slightly different text
> over time and across provider-side model updates. Counts should reproduce
> closely; tiny variation in a handful of borderline cases is expected. The
> collected `data/*.jsonl` response files are included so results can be
> **verified without re-spending API credits** (see Section 5).

## 3. Data

- `data/allsides_full.csv` — raw QBias / AllSides dump (source: Haak & Schaer,
  2023, QBias). Large (~14 MB); the public dataset can also be obtained from the
  original authors.
- `data/election_articles.csv` — QBias filtered to election-related topics.
- `data/sampled_articles*.csv` — the balanced 999-article samples (two batches)
  and the smaller cross-model samples.
- `data/framed_prompts*.csv` — questions wrapped in the three framing templates.
- `data/rag_responses_*.jsonl` — raw model answers, one file per model/batch.
- `data/hallucination_labels_*.jsonl` — judge verdicts (corrected judge).
- `data/coding_sheet.txt` — the RQ2 directional-coding sheet generated by
  `07_build_coding_sheet.py`; the 11 genuine cases were coded **by hand** on
  this sheet (left / right / neutral / unclear).
- `data/eu_pilot_results.jsonl`, `data/harder_task_probe_results.jsonl` — the
  two probes reported in the Discussion.

## 4. Reproducing the pipeline from scratch

Run from the project root, in order. Per-model variants differ only by API
endpoint / model name and input file.

```bash
# Stage 0-2: data prep
python scripts/01_filter_election_articles.py
python scripts/02_sample_articles.py          # + sample_batch2.py / sample_batch3.py

# Stage 1: question generation
python scripts/03_generate_questions.py        # + _batch2 / _batch3 variants

# Stage 2: framing
python scripts/04_apply_framing.py             # + _batch2 / _batch3 variants

# Stage 3: grounded answering (one script per model config)
python scripts/05_rag_answer_openai.py             # GPT-4o-mini, batch 1
python scripts/05_rag_answer_batch2_openai.py      # GPT-4o-mini, batch 2
python scripts/05_rag_answer_deepseek_batch2.py    # DeepSeek V3.2
python scripts/05_rag_answer_batch3_llama.py       # Llama 3.3 70B (Groq)
python scripts/05c_rag_answer_batch3_deepinfra.py  # Llama 3.1 70B (DeepInfra)
python scripts/05b_rag_logprobs.py                 # GPT-4o-mini logprobs (RQ3)

# Stage 4: hallucination judge (corrected judge) — one per model config
python scripts/06_judge_hallucination.py
python scripts/06_judge_hallucination_batch2.py
python scripts/06_judge_hallucination_deepseek.py
python scripts/06_judge_hallucination_batch3_llama31.py
python scripts/06_judge_hallucination_batch3_llama33.py
python scripts/06_judge_rq3_corrected.py           # RQ3 subset, re-judged
```

All answering/judging scripts are **resumable**: they skip
`(article_id, frame_type)` pairs already present in their output file, so an
interrupted run can be restarted safely.

## 5. Verifying each result (no re-running needed)

Because the response and label files are included, each thesis number can be
checked directly from the committed data:

| Thesis item | How to verify | Script |
|---|---|---|
| **Table 2** — corrected hallucination counts & χ²/p by frame | recomputes counts, χ², p, and the Wilson 95% CIs per model | `make_rq1_charts.py` |
| **Figure 2** — before vs after judge correction (GPT-4o-mini) | red/blue bars from batch-1 labels | `make_rq1_charts.py` |
| **Figure 3** — corrected rate, 4 models | grouped bars, all < 1.1% | `make_rq1_charts.py` |
| **RQ2 / Figure 4** — 11 genuine cases, direction | inspect the hand-coded `data/coding_sheet.txt` | `07_build_coding_sheet.py` (+ manual coding) |
| **Table 3 / Figure 5** — entropy regression, pre vs post | odds ratio, ROC-AUC, mean entropy | `rq3_regression.py`, `rq3_regression_corrected.py` |
| **Figure 6** — harder-task probe (0.17% vs 2.22%) | `data/harder_task_probe_results.jsonl` | `check_harder_task_probe.py` |
| **Figure 7** — EU pilot, 24/24 supported | `data/eu_pilot_results.jsonl` | `check_eu_pilot8.py` |

The `check_*.py` / `pull_*.py` / `inspect_*.py` scripts are the diagnostic tools
used during analysis (label distributions, gap checks, pulling flagged
transcripts). They are kept for transparency but are not required to reproduce
headline results.

## 6. Reproducing the judge-bug discovery (the core contribution)

This is the heart of the thesis and is fully reproducible:

```bash
python scripts/test_judge.py            # ORIGINAL judge — misclassifies honest
                                        # premise-correction as UNSUPPORTED
python scripts/test_corrected_judge.py  # CORRECTED judge — 5 stress cases,
                                        # all classified correctly
```

- `test_judge.py` contains the **original (buggy)** judge prompt. It passes four
  hand-built stress cases but mislabels the fifth (a model that correctly flags
  an unsupported framing phrase while answering the factual question correctly).
- The corrected prompt — with the added instruction that flagging an unsupported
  framing phrase counts as **SUPPORTED** — is what the `06_judge_*.py` scripts
  use for the final analysis.
- Diff the `JUDGE_PROMPT_TEMPLATE` string in `test_judge.py` against
  `06_judge_hallucination.py` to see the exact one-instruction fix.

## 7. Final thesis figures

The final figures in the thesis (`*_R.png`, Figures 2–7) are generated by
`scripts/thesis_pipeline.R`. Run it in R/RStudio from the project root and it
renders each chart plus a summary table.

The numbers behind those figures are also fully reproducible in Python from the
saved `data/` files (see Section 5): `make_rq1_charts.py` for the RQ1 counts,
chi-square tests and 95% CIs, and `rq3_regression_corrected.py` for the entropy
results. So the figures can be re-rendered in R, and every value in them can be
independently verified with Python alone.

## 8. Models: included vs excluded

- **Included in the final analysis:** GPT-4o-mini (OpenAI), DeepSeek V3.2
  (DeepInfra), Llama 3.1 70B (DeepInfra), Llama 3.3 70B (Groq).
- **Tested and excluded:** NVIDIA Nemotron 3 Ultra (chain-of-thought output
  overran the token budget) and Mistral (exploratory only). The
  `*_nemotron*` and `*_mistral*` scripts/data are retained for transparency but
  are **not** part of the reported results.
